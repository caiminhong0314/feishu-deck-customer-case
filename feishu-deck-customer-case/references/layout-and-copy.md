# 布局与文案执行规则

## 操作卡

- 产出：完成 `DESIGN-PLAN.md`；创建版本 8 的 `layout-plan.json` 与校验报告。
- 命令：`python3 scripts/validate_layout_plan.py --plan <layout-plan> --run-dir <RUN_DIR>`。
- 红线：只实现已确认的 story、recipe、指标、主导任务与机制转折。
- 红线：`evidence_composition` 只能引用 asset plan，不得复制字段。
- 红线：一个具名顶层 grid 管理全部纵向区域；轨道闭合 100%。
- 红线：主导证据不可读、内容重叠或容器越界属于 error。
- 红线：结构变化回写 DeckJSON/计划，不得只改最终 HTML。

## 1. 页面意图

布局前记录 page role、一句话结论、主导业务任务、主导叙事元素、主导证据、可降级细节和已确认 recipe。客户案例是演示海报：先表达业务观点，再用证据证明，不是文档摘要。

recipe 必须来自 `assets/contracts/layout-recipes.json`；内部几何可自适应。经典语法与列角色只见 `layout-archetypes.md`，指标语义只见 `source-and-story.md` §2。

## 2. 阅读顺序

空间顺序必须匹配业务逻辑：对比为问题 -> 机制 -> 结果；流程为触发 -> 判断 -> 行动；平台案例为共享基础 -> 并行应用 -> 价值。同级区域从左到右，内部从上到下。

卡片标题同一水平行；总结或机制流程位于证据上方。经典案例中结果带不是第四列，流程不得离开机制区。`process-first-v1` 只按 `layout-archetypes.md` 的例外使用。

## 3. 标题与框架

结论标题包含客户/场景、机制和最强来源结果；必须符合 Token 的单行上限。超限先删重复措辞，不得缩字号、换行、隐藏溢出或侵占安全区。

标题前不加英文眉题或客户元信息；默认不放来源页脚、客户 Logo 或重复字标。合规要求的说明进入底部安全区或演讲者备注。

## 4. 文案与现状呈现

痛点使用对仗短语，机制写动作而非产品名。只有真实线性工作流才使用步骤。

过去区域含 2-4 条独立障碍时用 `stacked-cards`；每项必须有独立表面、边界、padding 与间距。只是同一结论的短补充时用 `concise-list`；没有并列项用 `none`。

机制证据构图、节点映射与说明模式只见 `evidence-assets.md` §2/§7。`layout-plan.json` 写：

```json
{"evidence_composition":{"ref":"asset-plan"}}
```

布局校验器直接读取 asset plan，并用其中顺序与节点映射核对页面容器和渲染 DOM。

## 5. 指标文案

不得在布局阶段改变 L1/L2/L3、单位语义或证据状态。故事合同要求的顶部指标必须原样进入结果带；L2/L3 与 baseline 不得占位。结果带视觉实现只见 `visual-contract.md` §5。

每个突出数字应回答不同证明问题；估算值保留“预计/测算”，百分点不得写成百分比增长。

## 6. 证据主导构图

优先放大最强真实证据，只保留证明不同主张的辅助图。证据选择、裁剪、滚动、灯箱与说明只见 `evidence-assets.md`。证据不可读时返回素材阶段，不得发明布局例外。

## 7. 区域与容器

顶层区域为 Header、可选 Metrics/Context、Main、Bottom Safe。一个模块只属于一个区域；每个区域只用一种背景策略：`region-frame`、`card-group` 或 `open-band`。禁止厚重嵌套卡、跨区背景和同级边框重叠。

如果证据主导，流程与指标降为轻量信息带；如果流程/指标主导，仍保留一张明确的主导视觉证据。

## 8. 空间预算

全部数值只读 `assets/tokens/case-tokens.json`。顶层 grid 轨道精确闭合 100%，未使用的指标空间归还主证据。不得先绝对定位标题，再用独立 `top` 摆放正文；受基础 Deck 限制时也必须从同一计划起点计算并由渲染校验确认。

布局计划记录 recipe、流程位置、指标位置、主导元素、证据顺序引用、容器归属/几何、来源比例、标题对齐、总结位置和背景策略。

## 9. 横向分配

依据优先级、真实比例和 Token 范围分配。不得默认等宽或固定三列比例。主导任务对应区域应最宽，并为截图文字保留最低可读宽度。

## 10. 字体与密度

使用短展示语、紧凑标签和明确层级；字号与间距只读 Token。不要把标题、总结、流程、正文和说明压平成同一层级。避免孤字、短尾换行和重复前缀。

缩小字号前依次：删重复文案、回收多余间距、减少弱模块、扩大主导区域。不得用 `overflow:hidden`、负边距或 transform 掩盖碰撞。

## 11. 防重叠检查

检查标题到正文、正文到页脚、流程/总结到证据、证据到说明、相邻面板之间的分隔。`in-image` 说明与图片允许零间距，但完整证据组与外部内容仍须正常分隔。

出现重叠、紧贴、越界或大死区时，修改布局计划或 DeckJSON 后重渲染；warning 记录但不阻断，证据不可读与几何失败直接作为 error。
